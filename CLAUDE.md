## Autonomous content engine (scheduler + blog + stats + admin)

This is the largest subsystem added since `AGENTS.md` and is **entirely undocumented there**. It self-publishes an African news blog and a stats dashboard from an in-process cron.

- **In-process scheduler** (`src/lib/scheduler/`): zero-dependency, started **once** from `src/instrumentation.ts` (only when `NEXT_RUNTIME==='nodejs'` and `NEXT_PHASE!=='phase-production-build'`, wrapped in try/catch). A 60s tick (`index.ts`, `TICK_MS=60_000`, `setInterval` is `unref()`'d so it never keeps the loop alive) compares each job's persisted `last_run_at` against its most-recent-due time and fires if overdue — **self-correcting/catch-up**, so a missed window after downtime runs on the next tick. State (`cursor`, `last_run_at`, `last_status`, `runs`) lives in **local SQLite** `scheduler_state` (`state.ts`), so rotation position survives redeploys. Concurrency guard is an **in-memory `running` Set + `globalThis.__bokariSchedulerStarted`** — assumes a **single long-lived container**; it will double-fire across replicas/serverless (no cross-process lock). Disable with `BOKARI_SCHEDULER_ENABLED=false`.
- **Three jobs** (`jobs.ts`, all UTC): `discover-daily` = `dailyAt(4)` (refreshes the Discover feed → writes **Supabase** `discover_articles` via `refreshJob.ts`); `articles-rotate` = `everyHours(5)` (one draft per fire, walking `CATEGORY_ORDER`'s 6 beats → full tour in 30h, → **local SQLite** `articles`); `stats-weekly` = `weeklyAt(1,3)` (Mon 03:00, refreshes `africa_stats`). `everyHours(5)` is **wall-clock-slot based** (UTC hours 0/5/10/15/20), not "5h since last run". `runArticlesRotation` advances the cursor in a `finally` **even on skip/failure**, so a beat that finds no sources is skipped that cycle, never retried.
- **Blog generator** (`src/lib/blog/`): `generate.ts` gathers sources via `searchNews()`, sanitizes snippets (prompt-injection defense), prompts the chat LLM through `chatWithFallback` (`{temperature:0.3, maxTokens:3800}` — **no hard-coded model**; uses whatever `ai/config.ts` routes to), then a tolerant 4-stage parser (`extractFieldsLoose` handles the model writing `body` as raw unquoted Markdown — the historical "model abandoned" cause — and strips a dangling JSON brace; `generateArticleForCategory` strips `Introduction`/`Conclusion` heading lines anywhere). **Quality gates that cause a clean skip** (no filler): ≥2 fetched sources, snippets ≥40 chars, aggregator homepages skipped, body must cite **≥2 distinct** sources, title ≥8 chars, body ≥300 chars. **The LLM never emits a URL** — `stripModelLinks` removes any it does, then `backlinks.ts` deterministically injects internal + brand links (`BRAND_REGISTRY`: Dicken AI / Floo / ZeroName / Bokari), so fabricated slugs/404s are structurally impossible. Stored as `status='draft'` / `origin='auto'` in `articles` (`store.ts`). `articles.ts ensureSeeded()` lazily inserts `SEED_ARTICLES` (`origin='seed'`, published) once when the table is empty, so `/blog` is never blank.
- **Admin review queue** (`src/app/admin/articles/`, `src/app/api/admin/articles/{route,[id]}.ts`): the human-in-the-loop. `GET` lists by `?status=`, `POST {category}` generates on demand (`maxDuration=300`), `PATCH` applies edits + `publish`/`reject`/`unpublish`. Public `/blog` reads `status='published'` from the same SQLite table. **Admin = email allow-list, no DB role:** `src/lib/auth/admin.ts` `requireAdmin()` resolves the Supabase user and checks `BOKARI_ADMIN_EMAILS` (comma-separated, default `dicken.media2@gmail.com`). There is **no WhatsApp approval** anywhere despite the project-memory "vision" note — and the cadence is **5h, not 10h**.
- **Stats engine** (`src/lib/stats/`, page `/data` "L'Afrique en chiffres"): two coexisting data kinds. (1) ~25 **scalar** cards (`schema.ts` `STAT_DEFS`, seed values) that the weekly job can refresh — `update.ts` searches via the `searxng.ts` adapter, asks the LLM to extract one number at temp 0, and **only writes if it parses AND falls within the def's `[min,max]` bounds** (hallucination guard); stored/overridden in SQLite `africa_stats` (`store.ts`). (2) **static chart series** (`charts.ts`: UN WPP / IMF / AfDB / GSMA constants, never auto-updated — edit by hand). `DataCharts.tsx` is Recharts with a hard-coded brand palette. `page.tsx` mixes store-driven and hard-coded values (e.g. `value="+4,2 %"`), and uses its own `SOURCES` numbering separate from `charts.ts`'s `CHART_SOURCES`.
- **Manual cron trigger** (`src/app/api/internal/cron/[job]/route.ts`, GET+POST, `maxDuration=300`): a host-crontab fallback for the in-process scheduler, gated by `CRON_SECRET` (Bearer or `?key=`) — **closed (401) when unset**. ⚠️ Separately, `src/app/api/discover/refresh/route.ts` is **fully unauthenticated** and triggers the same expensive Discover refresh — anyone can call it.

## Authentication — rewritten (email + password via Supabase Auth)

The old WhatsApp-OTP / Kapso / Meta module described in prior docs **is deleted from source.** `src/lib/auth/whatsapp/` and all of `provider.ts`/`jwt.ts`/`otp-store.ts`/`kapso-client.ts`/`meta-client.ts` are gone, along with synthetic `${phone}@whatsapp.bokari.app` emails, the `WHATSAPP_PROVIDER` router, `resetProviderCache`/`resetKapsoClientCache`, and the webhook verifier. **Do not reference any of it.**

- **Email + password via Supabase Auth (GoTrue).** The real client is `src/lib/hooks/useAuth.tsx`, which calls `supabase.auth.signUp` / `signInWithPassword` / `signOut` / `onAuthStateChange` **directly** — it does *not* hit `/api/auth/*`. The `/api/auth/{login,register,me,logout}` routes also wrap Supabase Auth but are largely **redundant** (the client bypasses them). `register()` returns a "check your inbox" message when no session comes back — i.e. **instant login requires Supabase "Confirm email" OFF**, an external project setting (see memory `bokari-dev-env-and-ops`). `/api/auth/me` returns **HTTP 200 with `{user:null}`** when unauthenticated (never 401 — check the body); `/api/auth/logout` is a no-op stub (real logout is client-side).
- **Sessions are Supabase JWTs.** `src/lib/supabase/server.ts` `createServerClient(req)` reads the bearer from the `Authorization` header, else the `sb-access-token` cookie, and builds an anon-key client with that token. This is how every API route resolves the caller.
- **Dead code to ignore** (no importers in `src/`): `src/lib/auth/session.ts` (HMAC `bokari_session` cookie, `AUTH_SECRET`) and `src/lib/auth/crypto.ts` (PBKDF2). The SQLite `users` + `phone_otps` tables (`schema.ts`, `drizzle/0007`) and `users.authProvider`/`phoneWhatsapp`/`phoneVerifiedAt` columns are **vestigial** (live users live in Supabase). `package.json` still lists dead deps `@kapso/whatsapp-cloud-api` and `bcryptjs`/`@types/bcryptjs`. `AuthModal.tsx` and the old auth code point to `docs/TODO-kapso-cleanup.md`, which **no longer exists** — dangling reference.
- **Guest mode** (`src/lib/auth/guest.ts`, `/api/guest/track`): anonymous, **3 queries / 24h**, **local SQLite** `guest_sessions`, cookie `_bk_anon` (30d). `GUEST_DAILY_LIMIT=3` is duplicated in `guest.ts` **and** `useAuth.tsx` — keep in sync.
- **`country.ts`**: cookie-based (`_country`, default Senegal `SN`), 15 countries (`SN,CI,ML,BF,NE,TG,BJ,CM,GA,CG,CD,FR,NG,GH,KE`); `parseCfIpCountry` reads the Cloudflare `CF-IPCountry` header. **Turnstile** (`/api/turnstile/verify`) still exists, still **not wired to the frontend**, returns `{ok:true, verified:false, skipped:true}` when `CLOUDFLARE_TURNSTILE_SECRET` is unset. `rate-limit.ts` is a generic in-memory token bucket (not persisted, per-process).

## Social + YouTube search via Bright Data

Both reach the search pipeline through the **string-based `engines` convention** in `search.ts`, never via direct imports — renaming a network/engine string in one place silently breaks routing. Both share one `BRIGHTDATA_API_KEY` and both **default to a free, keyless fallback**, so the premium path is fully optional. **Bright Data uses a slow trigger→poll→download flow** (`BRIGHTDATA_POLL_INTERVAL_MS=4000`, `BRIGHTDATA_POLL_TIMEOUT_MS=45000`) meant for the async-research path; on the SSE hot path it falls back once agent timeouts fire.

- **Social** (`src/lib/social/`): per-network router (`provider.ts`, modeled on the old WhatsApp provider — module-scope cached, `resetSocialProviderCache()`). Each of `x`/`reddit`/`linkedin` resolves to `brightdata` (datasets v3) or `site` (injects a `site:` operator and delegates to the scraper). `brightdata` **degrades to `site`** unless a key + `gd_…` dataset id are set. `actions/socialSearch.ts` calls `searchSearxng(q, {engines:[network]})`; a network fires only when its source toggle **and** classifier boolean are both on (`discussions`→reddit legacy; defaults to `['reddit']`). `BOKARI_SOCIAL_SEARCH_ENABLED=false` is **not** a kill switch — it forces every network to the cheap `site` path.
- **YouTube** (`src/lib/youtube/` + `src/lib/agents/media/`): `engines:['youtube']` → `cachedYouTubeSearch` (provider router: `api` if `YOUTUBE_API_KEY` present, `brightdata`, or keyless `scrape`). The scrape provider re-enters via `engines:['youtube_scrape']` specifically to **avoid infinite recursion**. Comprehension (`media/transcript.ts` chain: Bright Data → youtubei.js InnerTube → managed API → ElevenLabs STT; `media/comprehend.ts` = chunk + BGE-M3 cosine top-k) does **no summarization** — it returns prompt-ready transcript windows with `[mm:ss]` markers; the writer LLM produces the prose. `actions/youtubeComprehend.ts` fires only when an actual youtube URL is in the query (no bare-id matching). Slow async path at `POST /api/youtube/comprehend` (202+jobId, GET polls, **same in-memory research-job store** as `/api/research-async`). ElevenLabs STT (African-language audio) is opt-in and **off** (`BOKARI_YOUTUBE_STT_ENABLED`). `/api/videos` is a separate **legacy** video-search endpoint, not the comprehension path.
- ⚠️ **`youtube_cache` is dead code.** It is declared in `schema.ts`, `drizzle/0009_youtube_cache.sql`, and `supabase/migrations/20260610_youtube_cache.sql`, but **no code reads/writes it.** Actual YouTube/social/transcript caching reuses the **`SemanticCache`** (`cache.sqlite`) with namespaced `youtube:`/`social:<net>:`/`transcript:` query-hash keys (exact-hash only, placeholder `[0]` embedding, TTL `FRESH_TTL_MS`=30min). See the migration-collision note below.

## Rich answer blocks + faithfulness badge

Structured visual blocks render alongside the streamed prose, in **two persistence classes** — this split is the central gotcha.

- **Transient (client-only, NOT persisted):** `chart` (`ChartSpec`), the three rich illustration blocks `comparison_table` / `entity_card` / `verdict` (`src/lib/agents/search/richBlocks.ts`), and the `faithfulness` NLI report. These are emitted as SSE `data` events (`index.ts`), merged into React state in `useChat.tsx`, and the DB write persists **only `session.getAllBlocks()`** — so after a refresh from Supabase they are **gone**, and they can **never** appear in public shares.
- **Persisted (`messages.response_blocks`):** `text`, `source`, `research`, and the Learn-mode `flashcard` / `quiz` blocks (`src/lib/types.ts`).
- **Opt-in gates:** rich blocks need `BOKARI_RICH_BLOCKS_ENABLED='true'`, faithfulness needs `BOKARI_FAITHFULNESS_ENABLED='true'` (**both default off**). **Charts have no gate** — they fire whenever `looksLikeChartRequest` matches. Every extractor **fails closed** (LLM/JSON error → null/empty, prose always stands). Trust gates are hard-coded in the coercers (e.g. a `verdict` needs a valid label in `{vrai,faux,trompeur,non_verifie}` and ≥1 source — no unsourced verdicts; verdict is pushed first as "the category-defining trust block").
- The renderer is **`src/components/MessageBox.tsx`** (there is no `MessageRenderer/index.tsx`). `FaithfulnessBadge.tsx` lives under `components/Message/`, renders an "X/Y affirmations vérifiées" pill, and only for logged-in users. `chartVariants.tsx` implements bar/line/area/pie/radar only — declared `scatter`/`composed` fall through to bar.

## Async deep research + observability

- **Async research** (`src/app/api/research-async/`, `agents/search/async.ts`, `src/lib/jobs/research.ts`): the `quality`/"Approfondie" mode (~35 iterations, 30–60s) runs as a background job to dodge the SSE timeout. POST → `202 + jobId`; client polls GET. The job store is an **in-memory `Map`** (`globalThis`-pinned, `MAX_JOBS=100`, pruned after 1h) — jobs don't survive a restart and won't work across serverless containers. Polling is **intentionally unauthenticated**: the 128-bit `jobId` is the bearer secret.
- **Observability** (`src/lib/observability/`): `latence.ts` is a stateless **millisecond** timer (`performance.now()`, not hrtime) that `console.warn`s per-stage; `ttfb.ts` is an in-memory LRU-bounded per-stage store exposing p50/p95/p99. **Per-stage**, not HTTP-request latency. See `docs/architecture/PERFORMANCE.md`.

## AI layer split: `src/lib/ai/` vs `src/lib/models/`

- `src/lib/ai/` = orchestration: `gateway.ts` (single entry for chat/embeddings + provider fallback), `config.ts` (env-driven routes; **BGE-M3 embeddings hardcoded to 1024 dims** — `BOKARI_EMBEDDING_DIMENSIONS` must match if you swap models or cosine breaks), `embedCache.ts` (in-memory LRU query→vector, disable via `EMBED_CACHE_DISABLED`), `reranker.ts` (cross-encoder; `live` OpenRouter vs `offline` deterministic — tests use offline; `BOKARI_RERANK_ENABLED='true'` for live).
- `src/lib/models/` = provider abstraction: `registry.ts` (lazy loader), `base/{llm,embedding}.ts`, `providers/` (openai, anthropic, groq, openrouter, ollama, gemini, lemonade, lmstudio, transformers).
- **Chat fallback** (`gateway.ts` `chatWithFallback`) fires on **two** paths: (a) the primary fails to **load**, and (b) the loaded primary **throws at call time**. Only a non-throwing but semantically-bad response is not retried. (Prior docs said "only on load throw" — wrong.)
- **Semantic cache** (`src/lib/cache/`): SQLite-backed (`store.ts`, better-sqlite3 WAL), query→response with a **0.90 cosine threshold** (`semantic.ts`, env-tunable via `BOKARI_CACHE_COSINE_THRESHOLD`, was 0.92). Float32 packing in `vector.ts`. Linear O(N) scan. Also the backing store for the YouTube/social/transcript caches above (distinct from `ai/embedCache.ts`).

## Things that will trip you up (added since AGENTS.md)

- **Duplicate `0009` migration prefix.** `drizzle/` has both `0009_autonomous_content.sql` and `0009_youtube_cache.sql`. The runner (`src/lib/db/sqlite.ts`) keys `ran_migrations` on `file.split('_')[0]` (just `"0009"`) and processes files alphabetically, so `autonomous_content` runs and records `"0009"`, then `youtube_cache` is **skipped** — its table is never created on a fresh DB. Harmless only because that table is dead code (above). **Never assume a migration number is unique; do not add another `0009`.**
- **`nanoid` is imported by `src/lib/learn/decks.ts` but is NOT in `package.json` dependencies** — it resolves only transitively today and could break on a clean install or dep bump.
- **`.env.example` is stale**: still names TinyFish, and omits `CRON_SECRET`, `BOKARI_ADMIN_EMAILS`, `BOKARI_SCHEDULER_ENABLED`, `BOKARI_LEARN_STORE` (all real runtime flags).
- **The whole autonomous pipeline assumes one long-lived container** (Hetzner Docker deploy) — the scheduler's in-memory guard, the research/comprehend in-memory job Maps, and the `unref()`'d interval all break under serverless/replicas.
