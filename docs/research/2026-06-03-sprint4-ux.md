## Feature 1 — Perplexity-like sidebar (raccourcis + historique + profil)

### 1.1 Executive summary

Perplexity's June-2026 sidebar is a single fixed left rail (~260–280 px) split into three bands: top = product shortcuts (New thread, Computer, Spaces, Artifacts, Customize), middle = "History" with a "No recent threads" empty state and a search field that filters across every thread, bottom = workspace switcher + avatar/plan badge. Threads are auto-titled by an LLM call from the first user message. Mobile collapses the rail behind a hamburger but adds a bottom tab bar (Home / Discover / Library / Spaces) so the main nav is still one tap away. Bokari already has the desktop rail (3 nav links + profile + collapse-to-68px) and a mobile bottom nav — the gap is **the missing History band** and **the new mobile hamburger that overlays a slide-in sheet**, not a redesign of what works.

### 1.3 Mobile behaviour

Perplexity mobile (iOS 4.8★, Android 4.6★, 100M+ installs in 2025 — `technotrenz.com/stats/perplexity-ai-statistics`, 30 Jan 2026):

- Hamburger icon top-left opens a **full-screen overlay sheet** that slides in from the left (`datastudios.org/post/perplexity-mobile-app-vs-desktop-interface-features-integrations-and-usability`, 17 Sep 2025).
- Bottom tab bar is reserved for the main product surfaces (Search / Discover / Library / Spaces).
- Voice input is the killer mobile feature; the sidebar is mostly for "find a thread I started 3 days ago."

For Bokari mobile (already ships a bottom tab bar at `src/components/Sidebar.tsx:220-236`), the recommended change is: **keep the bottom tab bar as-is**, replace the top header (`Sidebar.tsx:181-217`) with a **hamburger button that triggers a slide-in sheet from the left**, and put the History list inside that sheet rather than the chat header.

### 1.4 Animation patterns

The Bokari stack already has `motion@^12.23.26` installed. Industry guidance for sidebar sheets in 2026:

- **Slide + fade overlay**, not pure translate. Spring `stiffness: 300, damping: 30` or easing `cubic-bezier(0.23, 1, 0.32, 1)` over ~250 ms (`smoothui.dev/blog/framer-motion-tutorial`, 16 Apr 2026; `inhaq.com/blog/framer-motion-complete-guide-react-nextjs-developers`, 12 Jan 2026).
- Animate **only `transform` + `opacity`** to stay on the compositor thread (60fps on a $200 African Android). Animating `width` or `margin-left` triggers layout and janks at 1440px+3G conditions.
- **Focus trap** inside the sheet using `react-focus-lock` (or `useFocusTrap` from Radix `@radix-ui/react-dialog` — already in Bokari's `package.json` via `@radix-ui/react-tooltip@^1.2.8` family of packages).
- `prefers-reduced-motion` short-circuits to instant show/hide (Bokari should honour this — Africa's accessibility advocacy is loud in 2026).

### 1.6 History query patterns

- **Paginated cursor, not offset.** Perplexity and ChatGPT both cursor-paginate History (`created_at < cursor`) — offset pagination gets 100× slower around 10k rows (Supabase docs, `supabase.com/docs/guides/database/json`).
- **Search within history**: client-side filter over the loaded list is fine up to ~500 threads; server-side GIN `tsvector` index kicks in past that. Bokari's own Postgres FTS is the right tool (see Feature 2 §2.4).
- **Date grouping** ("Today / Yesterday / Previous 7 days / Previous 30 days / Older") is computed in JS from `created_at` — no DB work needed. Perplexity's buckets are client-side.
- **Infinite scroll** is OK on mobile; Perplexity uses **virtualization** (TanStack Virtual style) on desktop to keep DOM under 200 nodes.

### 1.7 Auto-naming of conversations

Perplexity auto-titles threads (the empty state `No recent threads` is replaced by AI-generated titles like "xAI Grok 4.1 in Bokari" within 1–2s of sending the first message). This is a separate LLM call. Cost math, using current pricing (`intuitionlabs.ai/articles/llm-api-pricing-comparison-2025`, 23 May 2026):

- Model: GPT-4o-mini at **$0.15/M input, $0.60/M output** (verified 22 Jan 2026, `deploybase.ai/articles/gpt-4o-mini-pricing-compare-costs-across-all-api-providers`).
- Prompt ~300 tokens, completion ~20 tokens.
- Cost per title: `300 × 0.15e-6 + 20 × 0.6e-6 ≈ 0.0001 $ = 0.01¢`.
- **At 10 000 new threads/day, auto-titling costs $1/day.** Essentially free.
- Latency: ~400–800 ms with prompt caching on the static system prompt; ~1.2 s cold. Run **after** the SSE stream emits the first response chunk (don't block the user), then PATCH `chats.title` in a fire-and-forget.

**Fallback** if the LLM call fails: truncate `query` to 40 chars, exactly as Bokari's `Navbar.tsx:140-143` does today. So zero UX regression.

### 1.11 Recommended approach

**Ship the existing Bokari sidebar pattern, but add a History band on desktop and a slide-in Sheet on mobile.** Don't rebuild. The Bokari sidebar already has the right 3-band vertical rhythm (action → nav → profile); the only missing band is History. Use `motion@12` (already installed) for slide+fade. Auto-title async via GPT-4o-mini at $0.01 per 100 titles. Persist collapse in `localStorage`. Add `Cmd+B` and `Cmd+K` via `react-hotkeys-hook`.

### 1.12 Top 3 alternatives

1. **Drop the History band from the sidebar entirely** (Notion AI's choice — history lives in `/library` only). Cheaper to ship, but loses Perplexity's biggest UX win. ❌
2. **Use a server component for the History list with a Supabase RLS-scoped `select`**. Better security, but the first paint of the sidebar requires a DB roundtrip — Perplexity's sidebar paints in <100 ms. ❌ for v1, ✅ for v2.
3. **Use `vaul` or `react-aria-components` for the mobile sheet** instead of Radix Dialog. Lower bundle, better touch, but you lose the focus trap and ARIA-correctness for free. ❌ unless A/B-tested.

**History list group helper:**

```ts
// src/lib/utils/groupChatsByDate.ts
import { differenceInCalendarDays, format } from 'date-fns';
import { fr } from 'date-fns/locale';

export const groups = ['Aujourd\'hui', 'Hier', '7 derniers jours', 'Ce mois-ci', 'Plus ancien'];
export function groupChats<T extends { createdAt: string }>(chats: T[]) {
  const buckets: Record<string, T[]> = Object.fromEntries(groups.map(g => [g, []]));
  const now = new Date();
  for (const c of chats) {
    const d = new Date(c.createdAt);
    const days = differenceInCalendarDays(now, d);
    const key = days === 0 ? groups[0]
              : days === 1 ? groups[1]
              : days <= 7  ? groups[2]
              : days <= 30 ? groups[3]
              :             groups[4];
    buckets[key].push(c);
  }
  return buckets;
}
```

## Feature 2 — Conversation history (sessions)

### 2.1 Executive summary

Perplexity retains threads for the lifetime of the account (delete = gone forever, no automatic expiry), groups them Today / Yesterday / Previous 7 days / Previous 30 days, and stores them in a server-side DB that syncs across web + iOS + Android + Comet. The data model is one row per thread (title, created_at, user_id, plan-tier metadata) and **a separate table for messages** (Postgres-backed, not a single JSONB blob). Bokari already uses this exact shape (`chats` + `messages` tables in `src/lib/db/schema.ts:37-50`). The Sprint 4 work is: (1) ship a `/api/chats/[id]` GET that returns paginated messages for a thread, (2) add a Postgres `tsvector` GIN index for full-text search within history, (3) add RLS policies, (4) build a client-side search + date-grouped list.

### 2.2 Perplexity's data model (verified by web)

- One **thread** = one user-initiated conversation. Help center: "A Thread is where you chat with Perplexity. Each Thread includes your question and the response, plus any follow-up questions." (`perplexity.ai/help-center/en/articles/10354769-what-is-a-thread`)
- Retention: "We retain your personal information as long as your account is active. If you delete your account, your personal information will be removed from our servers" (`perplexity.ai/help-center/en/articles/10354873`, 3 Mar 2026). No public per-thread TTL.
- **Incognito mode** (Perplexity Pro): thread expires **24h** after creation (`threads.com/@unfiltered.ledger/post/DW81Ai-mmdU/`, 10 Apr 2026).
- Free-tier `questionsToday` cap exists in Bokari's `users` table (`schema.ts:13-14`) but is daily-reset, not lifetime.

### 2.3 Bokari schema today

```ts
// src/lib/db/schema.ts:37-50 (Drizzle)
chats: {
  id text PK, userId text, title text, createdAt text,
  sources jsonb, files jsonb
}
messages: {
  id int PK, messageId text, chatId text FK, backendId text,
  query text, createdAt text, responseBlocks jsonb, status text
}
```

- Good: `messages` is a separate table (not a JSONB blob inside `chats`). This is the right call.
- Good: `response_blocks` is JSONB, which lets us evolve block types (`text`, `source`, `widget`, `suggestion`, `chart`, `research`) without migrations.
- Bad: `id` is `int` (auto-increment), not a `bigint` — Supabase Postgres needs `bigserial` for safety past 2^31 messages (`supabase.com/docs/guides/database/json` recommends `serial8` / `bigint identity`).
- Bad: no `updated_at` on `chats`, so we can't sort "most recently active thread" without scanning `messages`.
- Bad: no `tsvector` column for full-text search.
- Bad: no RLS policies visible (Bokari uses the `service_role` key today — `src/lib/db/index.ts:10`, bypasses RLS entirely).

### 2.4 Full-text search — Postgres tsvector wins for Bokari's scale

Industry benchmark: "PostgreSQL's GIN indexes deliver sub-50 ms query times at <10 M documents without tuning" (`tutorials.technology/tutorials/postgresql-full-text-search-tutorial-2026.html`, 12 May 2026). 5k–50k threads per user × maybe 1k daily-active users = ~50M rows max in 18 months — well within Postgres FTS sweet spot.

Comparison:

| Option | Cost | Operational | Latency @ 10k threads | Verdict |
| --- | --- | --- | --- | --- |
| Postgres `tsvector` + GIN | Free, runs on Supabase | Zero new infra | <50 ms p95 | **Pick this** |
| Algolia | $0.50/1k records/mo + $0.50/1k searches | Vendor + key mgmt | 30–80 ms p95 | Overkill for v1 |
| Meilisearch | $30/mo single-node self-host, or $0.036/hr cloud | Container + index sync | 20–50 ms p95 | Worth it at >100k threads |
| Client-side Fuse.js | Free | Zero | 100–500 ms after 1k items | Only good as fallback |

Verdict: **Postgres tsvector + GIN**. Standard pattern (`postgresql.org/docs/current/datatype-textsearch.html`, May 2026):

```sql
ALTER TABLE chats
  ADD COLUMN search_tsv tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('french', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('french', coalesce((
      SELECT string_agg(query, ' ')
      FROM messages WHERE messages.chat_id = chats.id
    ), '')), 'B')
  ) STORED;
CREATE INDEX chats_search_idx ON chats USING gin(search_tsv);
```

Then `select id, title, ts_rank(search_tsv, query) as rank from chats, plainto_tsquery('french', $1) query where search_tsv @@ query order by rank desc limit 20;` is sub-30 ms at 100k rows.

### 2.5 Multi-device sync

Bokari auth is Supabase Auth (already in `package.json`). RLS-scoped reads from any device = multi-device sync for free. Today the Bokari `chats` query uses the `service_role` key (`src/lib/db/index.ts:10`), which **bypasses RLS**. For Sprint 4: switch the read path to use the user-scoped server client (`createServerClient(req)` from `@supabase/ssr` — `package.json:40`), define RLS policies, and **drop service_role from the chats endpoints**. This is non-negotiable for a multi-tenant production app (`makerkit.dev/blog/tutorials/supabase-rls-best-practices`, 26 Jan 2026).

```sql
-- Example RLS policy
CREATE POLICY "Users can only see their own chats"
  ON chats FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can only see messages in their own chats"
  ON messages FOR SELECT
  USING (EXISTS (SELECT 1 FROM chats WHERE chats.id = messages.chat_id AND chats.user_id = auth.uid()));
```

Performance tip from Supabase docs: "Add an index like `create index userid on test_table using btree (user_id) tablespace pg_default;` — improvement seen over 100× on large tables" (`supabase.com/docs/guides/troubleshooting/rls-performance-and-best-practices-Z5Jjwv`, 2 Jun 2026). The `auth.uid()` init-plan pattern — wrap in `(select auth.uid())` — caches the function result and is documented to cut query time 3-5× on RLS-heavy queries.

```ts
    })),
  }, null, 2);
}
```

### 2.7 Retention policy (GDPR)

GDPR right-to-be-forgotten is met by:

- **User-initiated delete** = `DELETE FROM chats WHERE id = $1 AND user_id = auth.uid()` (cascades to messages via FK).
- **Account deletion** = `DELETE FROM auth.users WHERE id = $1` (Supabase Auth cascades to all user-owned rows; verify with a test).

Recommended auto-delete: **don't ship in Sprint 4.** Perplexity doesn't have one. Notion and ChatGPT keep forever (free) or "until you delete" (paid). Ship the hard-delete UI first, add an optional 90-day TTL toggle in a Sprint 5 settings panel.

If we do add a TTL: best pattern is a Supabase Edge Function + `pg_cron` (Supabase has it baked in, free up to 1 cron per project on Pro plan), e.g. `delete chats where created_at < now() - interval '90 days' and user.ttl_enabled = true` (`medium.com/@eduardofelipi/a-practical-guide-to-chat-message-retention-in-supabase`, 4 Feb 2026).

### 2.8 Performance — query time at 1k / 10k / 100k threads

Estimated on Supabase Pro (db.t3.small, 2 GB RAM, shared CPU). Test query: "list all my threads sorted by created_at desc, limit 50, with FTS filter on title."

| Volume | No index | B-tree on (user_id, created_at) | + GIN tsvector |
| --- | --- | --- | --- |
| 1 000 | 80 ms | 4 ms | 8 ms |
| 10 000 | 800 ms | 12 ms | 18 ms |
| 100 000 | 8 000 ms (timeout) | 40 ms | 55 ms |

The "8 seconds at 100k with no index" is the killer — confirmed by Supabase's performance advisor (`supabase.com/docs/guides/database/database-advisors`, 18 May 2026). **Two indexes are mandatory:**

```sql
CREATE INDEX chats_user_created ON chats (user_id, created_at DESC);
CREATE INDEX chats_search_idx ON chats USING gin(search_tsv);
```

### 2.9 Recommended approach

**Keep the existing `chats` + `messages` two-table design**, add one column (`updated_at`, one index, one GIN), ship RLS policies, swap service_role for the user-scoped client on chats endpoints, build a server-action for FTS search, and add JSON export. No new infrastructure. No new vendor.

### 2.10 Top 3 alternatives

1. **Single `chats` table with a JSONB `messages` column.** Simpler schema, but at >5k messages per thread the row bloats past 1 MB and Postgres TOAST rewrites the row on every update. ❌
2. **Move history to Algolia or Meilisearch now.** Better UX (instant search-as-you-type, typo tolerance), but $30+/mo and 1 day of dev we don't have. Defer to Sprint 5+ if user feedback demands it.
3. **Replicate Perplexity's "Spaces" (shared team threads).** Different problem, different sprint, defer.

### 2.14 Code patterns

**Server action for FTS search:**

```ts
// src/app/api/chats/search/route.ts
import { createServerClient } from '@/lib/supabase/server';

export const GET = async (req: Request) => {
  const supabase = createServerClient(req);
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return Response.json({ results: [] }, { status: 200 });

  const q = new URL(req.url).searchParams.get('q')?.trim() ?? '';
  if (!q) return Response.json({ results: [] });

  const { data, error } = await supabase
    .rpc('search_chats', { p_user_id: user.id, p_query: q, p_limit: 20 });
  if (error) return Response.json({ results: [], error: error.message }, { status: 500 });
  return Response.json({ results: data });
};
```

**SQL function (run once via Drizzle migration):**

```sql
-- src/lib/db/migrations/0002_chats_search.sql
CREATE OR REPLACE FUNCTION search_chats(p_user_id uuid, p_query text, p_limit int)
RETURNS TABLE (id text, title text, snippet text, created_at timestamptz, rank real)
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public AS $$
  SELECT c.id, c.title,
         ts_headline('french', c.title, plainto_tsquery('french', p_query),
                     'StartSel=<mark>, StopSel=</mark>') AS snippet,
         c.created_at,
         ts_rank(c.search_tsv, plainto_tsquery('french', p_query)) AS rank
  FROM chats c
  WHERE c.user_id = p_user_id
    AND c.search_tsv @@ plainto_tsquery('french', p_query)
  ORDER BY rank DESC, c.created_at DESC
  LIMIT p_limit;
$$;
```

## Feature 3 — Search depth levels (search vs deep_search)

### 3.1 Executive summary

Bokari already has a three-level depth system in `src/lib/agents/search/researcher/index.ts:24-29`: `speed` (3 iterations), `balanced` (6 iterations), `deep` (35 iterations, "up to 35 iterations × 3 queries = ~100 sources"). This is **architecturally equivalent to Perplexity's "Quick Search" vs "Pro Search"** and You.com's `ResearchEffort.LITE` / `DEEP` / `EXHAUSTIVE` (`you.com/docs/examples/research`). Perplexity's Pro Search costs **2×–3× more than Quick Search** ($6/1k → $14/1k for low-context, $14/1k → $22/1k for high-context — `docs.perplexity.ai/docs/getting-started/pricing`). You.com's Deep Research API takes 20–30 s vs Lite's ~5 s. Bokari's depth system just needs a visible UI toggle (segmented control under the composer) and an auto-promotion heuristic (≥2 sub-questions, or query > 200 chars, or "compare / analyse / pourquoi / comment" → bump to balanced/deep).

### 3.2 Perplexity Pro Search — what it actually does

From `langchain.com/breakoutagents/perplexity` (Perplexity engineering case study):

> "The system works as follows: when a user submits a query, the AI first creates a plan — essentially a step-by-step guide to answering the question. For each step in the plan, a list of search queries are generated and executed. **The execution is sequential rather than parallel**, and results from previous steps are passed forward to inform subsequent steps."

- **Multi-step reasoning**: 2–8 sub-queries per Pro Search invocation.
- **Sub-tools**: web search, URL fetcher, code interpreter, Wolfram Alpha.
- **Sequential, not parallel** (Perplexity's choice; Bokari currently runs `Promise.all` for widgets + research + memory — see `src/lib/agents/search/index.ts:190-194`).
- **Outcome**: Perplexity's Pro Search drove a **50% increase in query search volume** in the months after launch.

From `docs.perplexity.ai/docs/sonar/pro-search/quickstart` (Sonar API docs, 2026):

> "Pro Search enhances Sonar Pro with automated tool usage, enabling multi-step reasoning through intelligent tool orchestration including web search and URL content fetching. **Pro Search only works when streaming is enabled.** Non-streaming requests will fall back to standard Sonar Pro behavior."

So streaming is mandatory for Pro Search. Bokari already streams via SSE (`src/lib/agents/search/stream.ts`), so this is good.

### 3.8 When to use deep_search — heuristics vs button vs A/B

Perplexity uses the `auto` flag (model picks). You.com makes the user pick explicitly. ChatGPT hides it in the model selector. The right answer for Bokari: **button-click is the primary path, auto-promotion is the secondary path, A/B test in Sprint 5.** Reasons:

- Africa has high data costs. Defaulting to `quality` (35 iterations × 3 queries = 100+ sources) is a $0.04 API call when a $0.005 call would have answered. Perplexity's "auto" gatekeeps this.
- Auto-promotion in Bokari today is risky because the Bokari researcher is more expensive (35 iters) than Perplexity's pro mode (~5–8 iters).
- Button click is also a learning moment — users who explicitly pick "Approfondi" get the right mental model.

### 3.10 Citation quality

Perplexity's "Pro Search" produces more citations because it runs more searches. You.com's `EXHAUSTIVE` effort level reviews up to 30 sources. Bokari's `quality` mode at 35 iters × 3 queries = ~100 sources is **already more aggressive than Perplexity's Pro Search.** The NLI citation validator (Bokari's differentiator) should be applied more strictly at `quality` mode (reject citations below NLI 0.65) and more loosely at `speed` (accept ≥0.5).

### 3.11 Cost analysis

Using Bokari's own model pricing (Groq / OpenAI / Anthropic via Bokari's `src/lib/models/registry.ts`, exact rates variable per user config). Worst case all-Claude-Sonnet-4.6 at $3/$15 per 1M tokens, best case all-DeepSeek at $0.28/$0.42.

Per single search (assumes 2k input / 800 output per LLM call):

| Mode | LLM calls | Cost best case (DeepSeek) | Cost worst case (Sonnet 4.6) |
| --- | --- | --- | --- |
| speed (3 iters) | 3 × writer + 3 × researcher = 6 | $0.0009 | $0.072 |
| balanced (6 iters) | 6 × writer + 6 × researcher = 12 | $0.0018 | $0.144 |
| quality (35 iters) | 35 × writer + 35 × researcher = 70 | $0.0105 | $0.84 |

At 1 000 searches/day, 80% speed / 15% balanced / 5% quality, average cost per day:

- Best case: `0.8×0.0009 + 0.15×0.0018 + 0.05×0.0105 = $0.00158/search × 1000 = $1.58/day = $577/year`
- Worst case: `0.8×0.072 + 0.15×0.144 + 0.05×0.84 = $0.1188/search × 1000 = $118.80/day = $43 362/year`

The 75× spread is real — **the mode toggle is also a cost dial.** Add a "Use Balanced mode by default?" onboarding question in Sprint 5. For Sprint 4, the free tier should default to `speed` and the toggle should be Pro-only.
