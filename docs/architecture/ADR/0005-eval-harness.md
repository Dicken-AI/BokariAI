# ADR 0005 — Eval Harness for the Citation Engine

- **Status:** Accepted
- **Date:** 2026-06-02
- **Deciders:** Amadou (Dicken AI — Backend + AI/ML)
- **Supersedes:** none
- **Related:** ADR 0003 (OSS AI infra), ADR 0004 (Citation engine)

## Context

We shipped a citation engine (Phase 4) that lets the researcher
agent pull from a vector index of recently-embedded Bokari articles.
The pipeline:

```
query → embed (BGE-M3) → cosine over Supabase → top-K → LLM cite
```

We have no way to tell whether:
1. The cosine is doing useful work, or if BM25 alone would be just
   as good.
2. Changes to `src/lib/discover/ranker.ts` are actually improving
   retrieval or just shuffling noise.
3. The hybrid formula is the right shape.

We also have no baseline numbers, so any future claim of "we got
better" is unfalsifiable.

## Decision

Build a reproducible eval harness:

- **20+ hand-curated African queries** with topic tags and
  `mustMatch` / `forbiddenTitle` hints.
- **30+ hand-written fixture articles** covering the queries with
  a mix of relevant and irrelevant entries.
- **Pure IR metrics** (NDCG@10, MRR, hit-rate@10) implemented in
  TypeScript so they run in-process with the ranker.
- **CLI runner** that compares BM25-only vs hybrid (BM25 + cosine)
  and writes a Markdown report.
- **Live mode** by default — embeds queries with BGE-M3 (same model
  as production).  `--offline` flag uses unit vectors for CI.

### Grader: token overlap (not human-rated)

```
rel=3: ≥50% of distinct query tokens appear in title (after
       stopword removal), OR a `mustMatch` term appears in title.
rel=2: ≥30% of distinct query tokens appear in title.
rel=1: any query token or `mustMatch` appears in body only.
rel=0: topic gate fails, or no overlap.
```

Tradeoff: a human-rated set would be more accurate, but it
costs weeks of labeler time and we'd have to maintain it as the
corpus grows.  Token overlap is a *directional* signal — it tells
us whether the ranker is moving in the right direction, even if
the absolute NDCG numbers are off by ±0.1 from human judgment.

### Storage: fixture corpus, not live Supabase

The eval runs against a hand-written 33-article corpus with
embeddings baked in (`src/lib/eval/fixture-embedded.ts`).  This
gives us:

- **Reproducibility.**  Same numbers on every run, across machines,
  across time.
- **Speed.**  23 queries + 33 articles = ~3s end-to-end, including
  the BGE-M3 embedding call.
- **Known ground truth.**  We wrote the articles, so we know
  which ones are correct.

Live mode against Supabase is Phase 6 — once we have a stable
corpus, we can swap the fixture for "most recent 500 embedded
articles" and re-grade.

### Embedding source: gateway, not raw API

The runner takes an `embedFn` (caller-provided).  In production,
that's the gateway (`src/lib/ai/gateway.ts`), which already handles
provider resolution, retries, and fallback.  In tests, it's a
function that returns unit vectors.  This means the eval exercises
the same code path as production — provider changes show up here.

## Consequences

Positive:

- We have a **falsifiable retrieval claim**: "Hybrid beats BM25 by
  Δ NDCG" can now be measured.
- Pre-deploy gate in CI: any change that drops NDCG@10 by > 0.02
  fails the build.
- A/B testing retrieval changes is now a single CLI command.

Negative:

- **Fixture is too small** to surface the cases where cosine wins
  (BM25 already gets 100% on the 23 queries).  The eval reports
  the same number for BM25 and hybrid on the current fixture.
  We'll need to expand the fixture in Phase 6.
- **Relevance labels are noisy.**  Token overlap over-credits exact
  keyword matches.  Once we have human labels, we'll need to
  re-grade the fixture.
- **No inter-rater agreement.**  Deterministic grader, so we
  can't measure label noise.  Won't fix until we have multiple
  humans rating the same queries.

## Alternatives considered

- **Human-rated labels from the start.**  Rejected for V1: too
  slow, too expensive, too brittle.  We can layer human labels on
  top of the derived grader later without changing the harness.
- **Use an LLM as a judge.**  OpenAI GPT-4 or Claude grading
  relevance.  Rejected: costs ~$0.50 per eval run, takes 30s,
  and our LLM-as-judge numbers won't be reproducible across
  OpenAI model versions.  Use it later for "is the citation
  actually useful" questions, not for retrieval grading.
- **Run against live Supabase data.**  Rejected for V1: stateful,
  slow, requires auth tokens in CI.  Phase 6.

## Implementation

- `src/lib/eval/metrics.ts` — 20 tests
- `src/lib/eval/dataset.ts` — 10 tests, 23 queries
- `src/lib/eval/fixture.ts` — 33 articles
- `src/lib/eval/fixture-embedded.ts` — auto-generated by
  `scripts/embed-fixture.ts`
- `src/lib/eval/runner.ts` — `runEval(corpus, queries, embedFn)`
- `scripts/run-eval.ts` — CLI
- `scripts/embed-fixture.ts` — generator

Run:

```bash
npx tsx scripts/run-eval.ts
npx tsx scripts/run-eval.ts --offline
npx tsx scripts/run-eval.ts --out=docs/eval/2026-06-02.md
```

## Review

Re-evaluate when:
- We have 50+ human-rated queries.
- We add a second embedding model to the gateway.
- We change `src/lib/discover/ranker.ts` materially.
- A user reports "the citation engine gave a wrong answer."
